# Question Flow

## Why this document exists

This document explains, in simple terms, how questions are shown to learners in Learn and Practice:

- where questions come from
- what is generated by code vs stored in DB
- how sync affects progress
- what happens for correct/wrong answers
- what happens when a learner finishes all questions
- how spaced repetition is applied

This is written for non-coders and product/QA teams.

---

## Quick summary

1. **Table topics** (like `tables_1_5`, `tables_6_10`) are generated by app code, not read from DB content.
2. **Non-table topics** are read from the local `questions` table in the app DB.
3. **Global sync does not copy the full question bank** from server to local.
4. Sync mainly moves **user state** (reviews, stats, settings, badges), not content catalog.
5. Practice now follows an always-on review-priority stage flow with stage completion prompts.

---

## 1) Question source: Table topics vs other topics

### Table topics

For table topics, questions are generated in code.

- Example generated card: `7 x 8 = 56`
- IDs are deterministic (same table fact gets same id)
- Deck generation is immediate on device

What this means:

- No dependency on server content for table cards
- Same table topic key always produces the same base deck

### Non-table topics

For non-table topics, questions are fetched from local DB:

- source table: `questions`
- filtered by selected topic tree (topic + descendants)

What this means:

- The app reads local content catalog
- Practice can limit and stream questions from DB
- Learn now loads non-table questions in chunks (windowed), not all at once

---

## 2) What sync does and does not sync

### Sync does move

- review state (repetition, interval, next review, etc.)
- practice stats (correct/wrong attempts)
- settings (topic selection, toggles, device-scoped session state)
- badges

### Sync does not move

- full question content catalog (`questions` master content)

Important:

- If you ask “how 1000+ questions come from global DB to local DB,” current answer is:
  - They **do not** come through `/reviews/push` and `/reviews/pull`.
  - Those endpoints sync learner progress/state, not full content bank.

## 3) If there are more than 1000 questions

There are two different flows:

### A) Content loading in app (local DB)

- Practice uses batching through queue logic (small chunks)
- Learn also uses chunked loading for non-table topics (small windows)
- Table topics are still generated immediately in code as a fixed deck

### B) **Sync pipeline (what gets exchanged with server**)

- Sync is not sending full questions catalog.
- Sync sends learner state changes: reviews, stats, settings, badges.
- Review pushes (**updated review records** from local DB to global DB) are chunked (`pushChunkSize`, default `128` reviews per request)
- Pull uses `since_rev_id` and returns changed state since last pull, so only newer changes are fetched, not full history every time.
- Stats/settings/badges are included with sync state payloads (not full content catalog)

Operational note:

- Large learner history is handled incrementally by revision-based sync for reviews.

### How `rev_id` sends the next chunk (non-technical)

Think of `rev_id` like a **receipt number** for sync changes.

- Every review change gets a running number on the server (rev id).
- The app remembers: "I have received everything up to receipt X."
- Next time, app asks: "Send me only changes **after X**."

So the app never re-downloads the whole history every time.

Simple example:

1. Device has data up to rev id **1200**.
2. Server now has changes up to **1650**.
3. On pull, device asks with `since_rev_id=1200`.
4. Server returns the next set after 1200 (in manageable chunks).
5. Device saves them and updates checkpoint to latest received rev id.
6. Next pull starts from that new checkpoint, not from 0.

Why this matters:

- Faster sync on slow networks.
- Lower data usage.
- Safer resume if a sync is interrupted.
- Large histories are processed step-by-step instead of one heavy transfer.

## 4) Practice queue: how it works

Practice now uses an always-on **priority stage pipeline**.

Stage order (strict):

1. Wrong answered cards
2. Due cards
3. Unseen cards
4. In-progress cards
5. Recently mastered cards

Behavior:

- learner stays in current stage until stage cards are exhausted
- when stage completes, app shows completion message and a button to continue to next stage
- after final stage, app shows regular "Practice more"
- random mode (if enabled) randomizes order **inside** a stage, not across stages

Current values in code:

- Practice queue fetch batch size: **10**

Why this design:

- weak cards are surfaced first
- learner gets explicit stage-by-stage completion
- flow remains predictable for QA and learners

---

## 4A) Learn windowed loading: how it works

For non-table topics, Learn now follows a windowed pattern:

1. Load first chunk from local DB.
2. Restore learner position (by `cardId` first, then `cardIndex` fallback).
3. Apply smart ordering so weak/due/unseen cards are shown earlier and recently mastered cards are deferred.
4. As learner reaches the end of loaded cards, fetch next chunk, append, and re-apply smart ordering.
5. Continue until topic is fully exhausted.

Current value in code:

- Learn non-table chunk size: **120**

Why this helps:

- faster first render on large topics
- lower memory spike vs loading full topic at once
- better learner focus (less repetition of already-mastered cards)
- smoother behavior when topic size grows in future versions

## 5) What happens when learner answers

### Correct answer

- practice stats: `correct +1`, `attempts +1`
- review state is updated with chosen rating (good/easy/hard)
- next review time generally moves further out

### Wrong answer

- practice stats: `wrong +1`, `attempts +1`
- scheduler forces rating as `again`
- question returns very soon in review timeline (near-term retry)

### Skipped (Next without entering answer)

- if learner submits with empty input, it is treated as **skip**, not wrong
- practice stats do **not** change (`attempts` and `correct` stay same)
- session score does **not** change
- question is moved to the **end of current practice queue**
- learner gets the next question immediately

---

## 6) What happens when queue is completed

### Practice

If current session queue runs out:

- screen shows topic complete
- “Practice more” starts a fresh session for same selected topic
- review history remains stored (not erased)

### Learn

If Learn deck is exhausted:

- screen shows topic complete
- “Learn more” restarts from first card of same selected topic
- this is deck restart behavior, not review-history deletion

## 7) If learner completes all questions in a topic

“Complete all questions” has two meanings:

1. **Session completion**: learner reached end of current queue/deck.
2. **Mastery over time**: learner answered questions repeatedly across days.

In this app, "across days" uses scheduler intervals (not a vague concept):

- First successful review: next review after **1 day**
- Second successful review: next review after **6 days**
- Later successful reviews: next review grows by previous interval x ease factor
- If learner answers wrong (`again`): returns in about **10 minutes** and repetition resets

So mastery means a card continues to reappear on these spaced schedules over multiple calendar days.

After session completion:

- Practice can restart immediately
- Due-review logic will still bring mastered/needs-review cards back based on schedule

So completion does not mean “never see again.”

## 8) Spaced repetition: Learn vs Practice

This app uses two different learning loops:

1. **Practice loop** = long-term memory scheduling
2. **Learn loop** = in-session understanding and pacing

### Practice (full spaced repetition engine)

Practice keeps a scheduler record per question and updates it after each answer.

Stored state per question:

- `repetition`: how many successful reviews in a row
- `interval`: gap before next review (in days)
- `ease factor`: growth multiplier used for future intervals
- `next_review`: exact due timestamp
- `last_result`: last rating (`again`, `hard`, `good`, `easy`)

How interval changes by answer:

- If answer is wrong (`again`):
  - `repetition` becomes `0`
  - `interval` becomes `0.007` day (about **10 minutes**)
  - card is brought back very soon for retry
- If answer is correct with success rating (`hard/good/easy`):
  - first success: `interval = 1` day
  - second success: `interval = 6` days
  - later successes: `interval = round(previous interval x ease factor)`

Queue priority in Practice:

1. wrong answered cards
2. due cards
3. unseen cards
4. in-progress cards
5. recently mastered cards

This is the core long-term retention engine.

### Learn (session learning flow)

Learn is optimized for current-session study flow, not long-term scheduling.

Learn behavior:

- loads the selected topic in chunks/windows
- uses smart ordering for non-table topics:
  - weak or due cards first
  - then unseen cards
  - then learning-in-progress cards
  - recently mastered/not-due cards later
- restores and saves learner position
- supports in-session pacing/reordering
- helps user complete coverage of a topic smoothly

Important difference:

- Learn helps the learner understand content now.
- Practice decides when cards should reappear over coming days/weeks.

## 9) Global sync ON/OFF impact on question flow

### Global sync ON

- selection and session state are shared (as defined in sync rules)
- learner can continue from other device after sync

### Global sync OFF

- device-scoped session/selection state is isolated per registered device
- progress records still sync for shared reporting

## 10) Troubleshooting checklist (QA friendly)

If behavior looks wrong, check in this order:

1. Is selected topic a **table topic** or normal DB topic?
2. Is global sync ON/OFF?
3. Did sync run successfully (push + pull)?
4. Does local DB contain expected session/progress setting keys?
5. For practice, is queue depleted or topic actually empty?
6. For spaced repetition checks, compare `next_review` timestamps and ratings.

## 10A) Scoring and accuracy (user-facing behavior)

Practice shows three related numbers with different meanings:

### Attempts

- increases only when learner submits a non-empty answer
- skipped/empty submit does not increase attempts

### Correct

- increases when submitted answer matches the expected answer

### Accuracy

- calculated as:
  - `accuracy % = round((correct / attempts) x 100)`
- if attempts is 0, accuracy is shown as 0% to avoid divide-by-zero

### Session score

- starts at 0 for a new practice session
- +1 for each correct answer
- -1 for each wrong answer
- never goes below 0 (minimum clamped to 0)
- skipped/empty submit does not change score

Why this design:

- score reflects answer quality in current session
- accuracy reflects correctness ratio
- skip allows pacing without unfair penalty when learner intentionally defers a card
